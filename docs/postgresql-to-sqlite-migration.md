# PostgreSQL 到 SQLite 迁移说明

本文档描述的是 `csdm-lite` 这一 SQLite 分叉的迁移背景与设计取舍。

## 背景

CS Demo Manager 最早采用 PostgreSQL，是因为当时的数据层目标更偏向“分析型数据库”：

- 比赛、回合、击杀、道具、站位等数据量很大，且查询维度多。
- 统计页面和搜索页面依赖了不少复杂 SQL 能力。
- 早期实现更看重 SQL 表达能力和查询自由度，而不是桌面端零配置部署。

这套设计在功能快速扩张阶段是合理的，但随着产品定位逐渐稳定，它也暴露出明显的桌面应用成本。

## 之前为什么用 PostgreSQL

### 1. 查询能力强

最初的实现大量依赖 PostgreSQL 的高级能力，例如：

- `ILIKE` 做大小写不敏感搜索
- `FILTER` 做聚合条件统计
- `DISTINCT ON` 做“每组取一条”
- `ANY` / 数组相关表达式
- `to_char`、类型转换和时间格式化
- 更宽松和成熟的聚合、分组、连接写法

对一个需要做大量比赛统计和筛选的应用来说，这些能力能显著降低开发门槛。

### 2. 更像传统后端数据库

早期模型更接近“本地客户端 + 独立数据库服务”的思路。PostgreSQL 在这类模式下有几个明显优点：

- 事务和并发能力成熟
- 对复杂 schema 和迁移支持稳定
- 对分析型查询容错高
- 调试 SQL 时工具链丰富

### 3. 当时优先保证功能上限

在产品前期，优先级往往是先把复杂查询、统计页和搜索页做出来。PostgreSQL 允许我们先用最直接的 SQL 写法快速实现，而不必过早受限于更轻量数据库的方言差异。

## 现在为什么迁移到 SQLite

### 1. 产品本质是单机桌面应用

CS Demo Manager 的核心使用场景是“单个用户在本机管理自己的 demo 数据”。这与 PostgreSQL 擅长的多用户、服务化部署场景并不匹配。

SQLite 更符合当前产品形态：

- 数据库就是一个本地文件
- 不需要单独安装数据库服务
- 不需要初始化用户、密码、端口、权限
- 更适合随应用一起打包、升级和迁移

### 2. 安装和上手成本明显更低

PostgreSQL 对普通桌面用户的最大问题不是性能，而是门槛：

- 首次安装复杂
- 容易出现连接失败、权限、端口占用、服务未启动等问题
- 用户需要理解“应用”和“数据库服务”是两套东西
- 支持成本高，排障链路长

迁移到 SQLite 后，用户拿到安装包就能直接用，本地数据库自动创建，整体体验更像真正的桌面软件。

### 3. 维护成本更低

继续维护 PostgreSQL 方案会让代码库长期承受这些成本：

- 数据库方言耦合越来越深
- 测试和打包环境更复杂
- CI、本地开发、用户环境三套数据库行为难统一
- 功能开发时总要考虑额外的外部服务依赖

SQLite 把这些复杂度收回到了应用内部，更容易测试、分发和支持。

### 4. 当前数据规模对 SQLite 是可接受的

CS Demo Manager 虽然会存很多比赛数据，但典型使用方式依然是本机单用户、以读为主、写入以导入分析为主。这个负载模型与 SQLite 的优势区间是匹配的。

只要：

- schema 设计合理
- 索引完整
- 批量写入路径稳定
- 复杂 PostgreSQL 特性做兼容改写

SQLite 足以支撑当前桌面端需求。

## 迁移时的设计原则

这次迁移不是简单“换一个驱动”，而是把数据库层目标改成：

- 应用运行时完全不依赖 PostgreSQL
- 安装、启动、分析、搜索都默认基于 SQLite
- 能保留的功能尽量保留
- 强依赖 PostgreSQL 特性的功能，如果维护成本过高，可以简化或取消
- 测试要覆盖 SQLite 的真实行为，而不是只在类型层面兼容

## 迁移后得到的收益

### 对用户

- 安装更简单
- 启动更直接
- 不再需要额外部署数据库
- 数据备份和迁移更容易，只需要数据库文件

### 对开发和维护

- 本地开发环境更轻
- 自动化测试更稳定
- 打包链路更短
- 问题定位更集中，不再跨应用和数据库服务两层排障

## 代价与取舍

迁移到 SQLite 也不是“零代价”，主要代价在于：

- 需要改写 PostgreSQL 方言 SQL
- 某些复杂查询要换写法
- 少数高度依赖 PostgreSQL 特性的功能可能要降级或取消

这是一次有意识的产品取舍：牺牲一部分数据库方言便利，换取更低的使用门槛、更好的桌面端一致性，以及更可持续的维护成本。

## 结论

PostgreSQL 在项目早期帮助我们快速搭出了复杂的数据分析能力，这个选择在当时是合理的。  
但对今天的 CS Demo Manager 来说，SQLite 更符合“单机桌面应用、零配置、易分发、易支持”的产品方向。

因此这次迁移的核心目标不是“数据库降级”，而是让架构更贴近真实使用场景，让应用本身更完整、更容易交付给最终用户。
