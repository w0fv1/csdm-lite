# 上游同步与合并流程规范

## 1. 文档目的

本文档用于规范 `csdm-lite` 与上游仓库之间的同步方式，目标是：

- 持续吸收上游仓库中有价值的改动
- 避免重新引入 PostgreSQL 运行时依赖
- 保持 SQLite 版本的长期可维护性
- 为后续维护者提供可复制、可迁移、可追溯的操作流程

## 2. 上游仓库信息

- 上游仓库：`https://github.com/akiver/cs-demo-manager`
- 本仓库定位：`csdm-lite`，即 CS Demo Manager 的 SQLite 分叉版本
- 同步原则：采用人工发起、Codex 辅助分析、按 SQLite 约束选择性合入的模式，不采用机械式全量照搬

## 3. 同步策略

### 3.1 总体策略

每次发现上游仓库有新提交时，由维护者发起一次同步任务，再由 Codex 按既定规则完成分析、筛选、改写和验证。

同步目标不是“完全等同于上游”，而是“在不破坏 SQLite 分叉设计的前提下，吸收值得保留的上游改动”。

### 3.2 SQLite 分叉约束

同步过程中，以下约束始终优先：

- 不重新引入 PostgreSQL 运行时依赖
- 不恢复 PostgreSQL 式的数据库连接体验
- SQLite 版本默认自动打开本地数据库
- 新增 SQL 查询必须在 SQLite 上可运行
- 新增 migration 必须在 SQLite 上可执行
- 打包产物必须继续保证 `better-sqlite3` 的 Electron ABI 正确

## 4. 同步触发方式

当需要同步上游时，维护者只需向 Codex 发起明确指令，例如：

`开始合并上游新代码`

推荐在指令中补充以下信息：

- 上一次已同步到的上游提交
- 本次目标上游提交、tag 或日期
- 是否只同步某一类改动
- 是否要求重新打包验证

如果未提供额外信息，则默认：

- 以上游仓库最新 `main` 为目标
- 由 Codex 先确认当前已同步基线
- 由 Codex 自行分析哪些改动应当合入

## 5. 标准操作流程

### 5.1 确认同步范围

同步开始前，必须先确认本次同步区间：

- 当前 SQLite 分叉已对齐到的上游提交
- 本次准备同步到的上游提交

该信息必须以明确提交哈希记录，不能只使用“最新版”“最近一次”这类模糊描述。

### 5.2 获取并分析上游差异

在动手改代码之前，先由 Codex 对上游差异做分类分析。

所有提交、文件或改动主题，应归入以下类别之一：

- `直接合入`
- `改写后合入`
- `明确放弃`
- `暂缓处理`

### 5.3 分类标准

通常情况下可按以下规则判断：

- UI、文案、样式、非数据库逻辑、小型 bugfix：优先考虑直接合入
- SQL、migration、打包、启动流程、数据库初始化相关：优先考虑改写后合入
- 明显重新依赖 PostgreSQL 特性的功能：优先考虑明确放弃
- 改动面过大、冲突过多、短期收益不高：优先考虑暂缓处理

### 5.4 按主题实施合并

不建议将所有上游变更一次性混成一个大补丁。推荐按主题分批处理，例如：

- UI 与文案
- Demo 分析链路
- 搜索与统计逻辑
- 数据库 schema 与 migration
- 启动、打包与本地运行链路

按主题处理有助于：

- 降低冲突复杂度
- 提高问题定位效率
- 便于后续回滚单一主题变更

### 5.5 验证要求

每次同步完成后，至少执行以下验证：

- `npm run compile`
- `npm test`
- `npm run package:dir`
- 启动应用
- 使用 `9207927388930273548_0.dem` 做烟雾测试

如果本次同步涉及启动、打包或数据库初始化逻辑，额外执行：

- 免安装版启动验证
- 安装包启动验证
- SQLite 自动打开数据库验证

## 6. 记录要求

### 6.1 为什么必须记录

同步记录是长期维护的基础。缺少记录会导致以下问题无法回答：

- 当前 SQLite 分叉已对齐到上游的哪个提交
- 某个上游修复是否已经同步
- 某段差异是 SQLite 分叉的有意保留，还是尚未处理

### 6.2 记录文件

所有同步基线和同步记录统一维护在：

- [upstream-sync-state.md](C:/Users/wofbi/Desktop/cs-demo-manager/docs/upstream-sync-state.md)

### 6.3 每次同步后必须更新的信息

每次同步完成后，必须更新以下内容：

- 上游仓库地址
- 上一次同步基线
- 本次目标提交
- 最终实际对齐到的上游提交
- 直接合入的内容摘要
- 改写后合入的内容摘要
- 明确放弃的内容摘要
- 暂缓处理的内容摘要
- 验证结果

## 7. 推荐的 Codex 指令模板

### 7.1 标准模板

`开始合并上游新代码。请先确认 docs/upstream-sync-state.md 中记录的当前上游基线，再分析从 <old_sha> 到 <new_sha> 的上游变更，按“直接合入 / 改写后合入 / 明确放弃 / 暂缓处理”分类，并按 SQLite 版本可长期维护的方式逐步合并。完成后更新 docs/upstream-sync-state.md 并记录验证结果。`

### 7.2 当旧基线未知时

`开始合并上游新代码。请先确认当前仓库在 docs/upstream-sync-state.md 中记录的上游基线，再继续分析并合并到上游最新 main，最后更新同步记录。`

## 8. 适用范围

本文档适用于：

- 当前仓库
- 当前 SQLite 分叉维护模式
- 后续任何需要同步上游仓库改动的维护场景

如果未来分叉策略发生变化，应同步更新本文档和同步状态文件。

## 9. 结论

本仓库采用的是“人工发起、Codex 辅助、SQLite 约束优先”的上游同步模式。

该模式不以零冲突为目标，而以以下结果为目标：

- 同步范围清晰
- 处理理由明确
- SQLite 约束稳定
- 每次同步均可追溯

只要同步记录持续维护，该模式可以长期支撑 SQLite 分叉版本吸收上游更新。
